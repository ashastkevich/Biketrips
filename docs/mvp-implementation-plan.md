# BikeTrips: план реализации MVP

## Цель MVP

Запустить mobile-first web-сервис, где организатор может создать велопоездку и получить публичную ссылку, а участник может найти ближайшую поездку, открыть карточку и записаться через веб или Telegram без сложной регистрации.

MVP должен закрывать два ключевых сценария:

- организатор быстро публикует поездку, управляет участниками и отправляет обновления;
- участник находит подходящую поездку, понимает условия и записывается без долгой переписки в чатах.

## Состав MVP

В первую версию входят:

- лента ближайших поездок;
- фильтры по городу, дате, уровню, темпу, типу велосипеда и формату поездки;
- публичная карточка поездки;
- форма создания поездки;
- запись участника;
- лимит мест и лист ожидания;
- кабинет организатора для управления поездкой и участниками;
- Telegram-уведомления о записи, изменениях, отмене и напоминаниях;
- базовая админка или служебный интерфейс для модерации.

За пределами MVP:

- полноценное мобильное приложение;
- сложная система рейтингов и отзывов;
- платежи;
- продвинутые клубные подписки;
- автоматический импорт поездок из внешних чатов;
- сложная работа с GPX и картами, если это замедляет запуск.

## Этап 6. Telegram-бот

Результат: Telegram используется для подтверждений, напоминаний и уведомлений.

Команды MVP:

```text
/start
/my_trips
/help
```

Сценарии:

- пользователь записался на поездку, бот отправляет подтверждение;
- организатор изменил время, место или маршрут, участники получают уведомление;
- организатор отменил поездку, участники получают уведомление;
- за 12-24 часа до старта участники получают напоминание;
- место освободилось, следующий участник из листа ожидания получает уведомление.

Задачи:

- создать приложение `apps/bot`;
- подключить Telegram Bot API;
- реализовать обработку `/start`;
- связать Telegram account с пользователем;
- подключить bot worker к API или typed API client;
- реализовать отправку notification jobs из очереди;
- добавить retry-логику и логирование ошибок отправки.

## Критерии готовности MVP

MVP считается готовым, если:

- организатор может создать и опубликовать поездку;
- у поездки есть публичная ссылка;
- участник может найти поездку в ленте;
- участник может открыть карточку и записаться;
- лимит мест работает корректно;
- при переполнении лимита участник попадает в лист ожидания;
- организатор видит список участников;
- организатор может отменить поездку или изменить важные детали;
- участники получают Telegram-уведомления об изменениях;
- есть базовая защита от редактирования чужих поездок;
- проект можно поднять локально через Docker Compose;
- ключевые сценарии покрыты тестами.

## Риски и решения

| Риск | Как снизить |
| --- | --- |
| MVP расползется из-за карт, GPX и мобильного приложения | Оставить карты и mobile app за пределами первой версии |
| Telegram auth усложнит старт | Разрешить запись через простую web-форму, Telegram подключить как улучшение сценария |
| Организаторы будут создавать неполные карточки | Сделать обязательными только критичные поля, остальные оформить как подсказки |
| Лимит мест и лист ожидания дадут гонки при записи | Использовать транзакции на backend |
| Уведомления будут теряться | Использовать Redis + BullMQ с retry и логированием ошибок |
